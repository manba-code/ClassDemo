## 🔴 Critical（必须修复）

### 1. 存储型 XSS：检索页使用 `v-html` 渲染用户内容

| 项 | 内容 |
|----|------|
| **问题类型** | Critical |
| **位置** | `src/views/knowledge/KnowledgeSearchView.vue:210-230` |
| **问题描述** | `highlightText()` 将关键词包裹为 `<mark>` 后，通过 `v-html` 渲染 `tags`、`snippet`、`fullContent`。知识点 `content` 来自数据库且 API 无鉴权，攻击者可写入 `<script>` 或事件属性，在其他用户浏览检索结果时执行。 |
| **影响** | 跨用户脚本注入，可窃取 Cookie、篡改页面或发起进一步攻击。 |
| **建议** | 优先对原始文本做 HTML 转义，再仅对关键词插入 `<mark>`；或引入 DOMPurify 白名单净化。检索高亮改为纯文本 + CSS 更安全。 |
| **是否处理** | 否 |

### 2. 知识图谱 Tooltip 未转义 HTML

| 项 | 内容 |
|----|------|
| **问题类型** | Critical |
| **位置** | `src/views/knowledge/KnowledgeGraphView.vue:62-67` |
| **问题描述** | ECharts `tooltip.formatter` 将 `d.content` 直接拼入 HTML 字符串。恶意内容可突破 tooltip 上下文执行脚本。 |
| **影响** | 与上条相同，攻击面在图谱页。 |
| **建议** | 对 `d.name`、`d.content` 做 HTML 实体转义后再拼接；或改用自定义 Vue 浮层渲染纯文本。 |
| **是否处理** | 否 |

### 3. 写操作 API 无鉴权

| 项 | 内容 |
|----|------|
| **问题类型** | Critical |
| **位置** | `KnowledgeController.java`、`LayerController.java` 全部 POST/PUT/DELETE 端点 |
| **问题描述** | 知识点与分层元数据的增删改接口对任意调用方开放，无 Token、Session 或角色校验。 |
| **影响** | 公网部署时任何人可篡改教学内容或清空数据；与 XSS 组合风险更高。 |
| **建议** | 至少增加教师端简单鉴权（API Key / JWT + 角色）；只读接口可对匿名开放。课程演示可保留现状，但需在 README 标明「仅限内网/本地」。 |
| **是否处理** | 否 |

## 🟡 Warning（建议修复）

### 4. 数据库凭据硬编码于配置文件

| 项 | 内容 |
|----|------|
| **问题类型** | Warning |
| **位置** | `backend/src/main/resources/application.yml:16-17`、`backend/docker-compose.yml:11-14` |
| **问题描述** | `username`/`password` 明文写在仓库中。 |
| **影响** | 仓库泄露即数据库暴露；不符合 12-Factor 配置原则。 |
| **建议** | 使用 `spring.profiles.active` 区分 `dev`/`prod`；生产环境通过环境变量注入 `SPRING_DATASOURCE_PASSWORD`。 |
| **是否处理** | 否 |

### 5. `spring.sql.init.mode: always` 不适合生产

| 项 | 内容 |
|----|------|
| **问题类型** | Warning |
| **位置** | `application.yml:22-27` |
| **问题描述** | 每次启动都执行 `schema.sql` 与 `seed.sql`。虽使用 `CREATE TABLE IF NOT EXISTS` 和 `INSERT IGNORE`，但仍会在每次启动连接数据库并尝试初始化。 |
| **影响** | 生产环境可能产生非预期行为；多实例并发启动时存在竞态。 |
| **建议** | 开发环境 `always`，生产环境 `never` 并改用 Flyway/Liquibase 做版本化迁移。 |
| **是否处理** | 否 |

### 6. 跨层检索全量内存分页

| 项 | 内容 |
|----|------|
| **问题类型** | Warning |
| **位置** | `KnowledgeSearchService.java:45-74` |
| **问题描述** | 遍历五张表 `selectList` 将全部匹配载入内存，再在 Java 中 `subList` 分页。 |
| **影响** | 数据量增大后内存与响应时间线性恶化；当前课程规模可接受，但扩展性差。 |
| **建议** | 各层分别 `COUNT` + 带 `LIMIT` 查询，或使用 UNION ALL + 数据库分页；短期可对 `keyword` 长度与总结果数设上限。 |
| **是否处理** | 否 |

### 7. 动态分表上下文缺失时的静默降级

| 项 | 内容 |
|----|------|
| **问题类型** | Warning |
| **位置** | `MybatisPlusConfig.java:23-28` |
| **问题描述** | `KnowledgeTableContext.get()` 为 `null` 时回退到逻辑表名 `knowledge_point`，而 schema 中不存在该物理表。 |
| **影响** | 若 `ThreadLocal` 未正确设置（异步、线程池复用等），将产生难排查的 SQL 异常。 |
| **建议** | 上下文为空时直接抛出 `IllegalStateException`，而非回退到不存在的表名。 |
| **是否处理** | 否 |

| **建议** | 对更新 DTO 增加 `@NotBlank`（配合可选字段策略）或自定义 `@AtLeastOneNotNull` 校验器，统一在 Controller 层拦截。 |

## 分模块简要评价

### 后端架构 ✅

`KnowledgePointRouter` 分表路由封装得当，调用方无需关心物理表名，符合单一职责：

```java
public <T> T execute(String layerKey, Function<KnowledgePointMapper, T> action) {
    LayerKey layer = resolve(layerKey);
    try {
        KnowledgeTableContext.set(layer.getTableName());
        return action.apply(knowledgePointMapper);
    } finally {
        KnowledgeTableContext.clear();
    }
}
```

### 异常处理 ✅

`GlobalExceptionHandler` 覆盖校验异常、业务异常、兜底 500，与 `ApiResponse` 统一包装一致。

### 前端 API 层 ✅

`client.js` 统一解析 `{ code, message, data }`，错误传播清晰；Vite 代理配置正确。

### 测试质量 ✅

Service/Controller 覆盖率 95%+，Given-When-Then 结构规范，符合项目测试规则。

## 总结

本项目作为《网络实习》课程演示系统，**后端分层与分表设计成熟，单元测试覆盖率达优秀水平，API 契约与 PRD 对齐度高**。主要短板集中在**安全边界**（XSS、无鉴权）和**环境配置隔离**（凭据、SQL 初始化策略）。

### 优先行动项（按顺序）

1. 修复检索页与图谱页的 HTML 注入（Critical）
2. 为写接口增加最小鉴权，或明确标注仅本地使用（Critical/Warning）
3. 拆分 `dev`/`prod` 配置，凭据环境变量化（Warning）
4. 删除 `KnowledgeSearchService` 无用依赖，加固动态分表空上下文处理（Warning）
5. 检索分页改为数据库层实现（数据增长后再做）
